# MindWeave Todo

本文档用于记录当前阶段的工程改进任务，便于落盘追踪与协作。

## 优先级定义

- P0：安全/数据风险，必须优先处理
- P1：核心价值闭环/关键体验
- P2：稳定性与可维护性增强

## 记录口径

- “Done”标准：代码合并后可在本地按 README 启动并复现验证；关键路径有至少 1 条自动化测试或可重复的手工验收步骤
- 影响面：优先描述对“镜像数据安全 / 可追溯 / Agent 自动化闭环”的影响，而非 UI 细节
- 约束：禁止向量化；所有设计以 Agent 可自动化为第一原则（见 AGENTS.md / prd.md）

## 任务清单（详细）

### P0

- [ ] 为同步实现冲突检测与覆盖保护
  - 背景/问题
    - 当前同步会对每个可转换源文件直接覆盖写入镜像 `.md`，会把 Agent 在镜像里做过的修改冲掉，且无备份
    - 这与“镜像可被 Agent 修改/可回滚”的目标冲突，属于数据丢失风险
  - 目标
    - 同步不再无条件覆盖；对“源文件变化”和“镜像被改”之间的冲突做可解释、可回滚的处理
  - 范围
    - 影响 `syncWorkspaceFiles()` 同步流程与镜像写入策略
  - 方案建议（择一或组合）
    - 最小可行：覆盖前对旧镜像生成 `.bak`，并记录冲突提示
    - 推荐：维护一份 `.mindweave/sync-state.json`（按源文件路径记录：源 mtime/size/hash 与镜像 hash），同步时判断：
      - 仅源变更：安全覆盖并更新状态
      - 仅镜像变更：跳过覆盖并提示“请人工处理/导出 patch”
      - 两边都变更：写入冲突副本（如 `xxx.conflict-YYYYMMDD-HHMM.md`）并提示
    - 可选增强：对纯文本（md/txt）做三方合并（base=上次同步快照，local=镜像当前，remote=源当前）
  - 验收标准
    - 同步后，用户在镜像里做的修改不会被静默覆盖
    - 发生冲突时可在镜像目录中找到备份/冲突副本，并能从 UI/提示信息理解发生了什么
    - 至少新增 1 个单测覆盖“镜像被改 + 源文件变更”的行为



### P1

- [ ] 在 UI 显式展示回答 Sources 并可复制
  - 背景/问题
    - 运行时会在 done chunk 中带回 `sources`，但当前 UI 主要展示内容与“Thinking Process”，用户看不到可核验的来源列表
  - 目标
    - 每条 assistant 消息在内容区下方展示 Sources（可折叠、可复制、可点击打开文件预览）
  - 交互建议
    - 空 sources：显示 “Sources: (none)”
    - 有 sources：展示为列表；点击后打开右侧 File Preview（如果是镜像路径）
    - 支持“一键复制 sources 列表”
  - 验收标准
    - 任意包含 sources 的回答都能在 UI 中看到来源列表，并能复制
    - 点击来源能打开对应文件（若路径可读）

- [ ] 为回答来源规则增加服务端强制校验
  - 背景/问题
    - 目前来源规则主要靠 system prompt 约束；模型可能遗漏 “Sources:” 或写出与实际读取不一致的来源
  - 目标
    - 让“可追溯”变成系统能力：至少保证“展示的 sources 来自真实工具调用”，并为用户提供一致的证据入口
  - 方案建议
    - 服务端在最终输出（done 之前）若检测到回答文本未包含 Sources，则追加：
      - `Sources:` + runtime 收集到的 sources（文件路径去重）
    - UI 不依赖模型文本里的 Sources，而是优先渲染结构化字段 `sources`
  - 验收标准
    - sources 字段与 UI 展示始终来自真实工具调用，不依赖模型自述
    - 回答文本可选追加 Sources，但不作为唯一来源展示方式

- [ ] 接入真实文档转换链路并产出可问答 Markdown
  - 背景/问题
    - `.pdf/.docx/.pptx/.xlsx` 当前仅生成占位镜像，无法实现“对话问答原始资料”的核心价值
  - 目标
    - 同步后这些格式生成可问答的 Markdown（至少文本可检索，尽量保留标题层级/表格）
  - 方案建议
    - 推荐方案：引入基于 Python 的专业大模型解析库（如 Microsoft `markitdown` 或 IBM `Docling`），专门为 Agent/LLM 场景设计，保留表格和排版层级。
    - 落地方式：在项目中维护一个轻量级的 Python CLI 脚本，Node 端通过 `child_process` 子进程调用进行异步转换。
    - 增量机制：利用源文件的 `mtime` 或 `hash` 记录状态，实现增量转换，避免每次同步重复解析。
    - 元数据与降级：转换产物顶部注入元数据（路径、时间等）；大文件截断保护上下文；转换失败时回退到带错误说明的占位文件。
  - 验收标准
    - 对上述 4 类文件至少各挑 1 个样例能转换出非占位 Markdown，并能被 search_files 命中
    - 失败时给出可诊断信息（错误原因、建议动作），不 silent fail

- [ ] 提供 Agent 修改文件的可视化 Diff 视图
  - 背景/问题
    - Agent 修改文件后仅提示“已修改”，缺乏直观的变更展示，用户难以建立对自动化操作的信任
  - 目标
    - 在对话流或 UI 中提供轻量级的 Diff 视图，让用户一眼看出改了什么
  - 验收标准
    - 触发文件修改时，回复消息或文件预览中能直观对比修改前后的差异

- [ ] 实现动态破冰引导与上下文感知（Suggested Prompts）
  - 背景/问题
    - 导入 Workspace 后，用户面对空输入框可能不知道如何提问，存在“白板综合征”
  - 目标
    - 根据当前目录下的文件特征，动态在输入框上方生成 2-3 个建议 Prompt
  - 验收标准
    - 切换 Workspace 或打开空对话时，能看到基于当前工作区内容的建议问题

- [ ] 原始文件与 Markdown 镜像的联动体验
  - 背景/问题
    - Markdown 转换会丢失排版，用户在核对复杂表格或图表时需要查看原文件
  - 目标
    - 在预览 Markdown 镜像时，提供快捷方式直接调用系统默认程序打开对应的原始文件
  - 验收标准
    - 文件预览区有“打开原文件”的入口，点击可拉起系统默认程序查看源文件

### P2

- [ ] 为 Agent 工具调用增加熔断与预算控制
  - 背景/问题
    - 当前聊天循环只要模型继续触发工具就会继续，缺少最大轮次/最大耗时/重复调用检测，存在“工具死循环”风险
  - 目标
    - 为一次对话引入可控预算：最大轮次、最大工具调用数、最大运行时长，并在触发时优雅停止
  - 方案建议
    - Budget：`maxRounds/maxToolCalls/maxMs`（默认保守值，可配置）
    - Repetition guard：对相同工具+相同参数的重复调用做计数，超过阈值停止
    - Stop UX：触发熔断时返回明确提示（原因 + 建议如何缩小问题或同步数据）
  - 验收标准
    - 人为构造工具循环时系统可停止并返回明确原因
    - 不影响正常“多轮工具链”任务完成

- [ ] 更新 PRD 以对齐当前实现与约束
  - 背景/问题
    - PRD 内部分技术栈与实现不一致（例如架构栈描述），且转换/来源等规则描述与当前实现存在偏差
  - 目标
    - 让 PRD 成为“能指导开发与验收”的文档：目标/范围/约束清晰，且以当前实现为准
  - 工作内容
    - 更新技术架构章节：Node/Electron、本地 HTTP+SSE、工具系统、非向量检索（grep-like + PageIndex）
    - 明确同步规则与冲突策略（与 P0 同步任务对齐）
    - 明确来源输出的系统保障与 UI 呈现（与 P1 对齐）
  - 验收标准
    - PRD 中关键链路描述（workspace→sync→search/read→chat→sources）可直接映射到仓库实现

- [ ] 完善 Agent 视野的“具象化”（Knowledge Snapshot）
  - 背景/问题
    - 用户不知道 Agent 能看到哪些文件，哪些文件因为格式不支持被忽略
  - 目标
    - 在 UI 中清晰展示当前镜像目录的有效文件数、忽略文件等状态，管理用户预期
  - 验收标准
    - 界面上有可查看的 Workspace 索引状态或文件解析概览面板

- [ ] 交互式卡片的降级处理（人在回路）
  - 背景/问题
    - 遇到冲突（如 P0 提到的文件冲突）或死循环熔断（如工具 Budget 耗尽）时，纯文本报错体验生硬
  - 目标
    - 将冲突处理和熔断拦截转化为可交互的 Action Card（例如“保留本地/采纳Agent”、“是否继续深挖”），把控制权交还给用户
  - 验收标准
    - 发生特定边界情况时，UI 渲染为包含明确行动按钮的卡片，点击后能触发后续处理
