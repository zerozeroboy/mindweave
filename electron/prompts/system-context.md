# MindWeave 系统上下文说明

## 1. 项目背景

MindWeave 是一个基于 Electron + React + TypeScript 的本地 AI 代理应用，旨在通过**编织散落的知识**来构建下一代文档交互方式。  
目标不是做“通用聊天”，而是做“面向本地文件工作流的可控助手”：

- 通过工作空间（workspace）机制，将用户源目录同步到镜像目录。
- AI 默认只在镜像目录内读写，降低误改源目录的风险。
- 以工具调用（`list_files` / `search_files` / `read_file` / `write_file` / `create_file`）为核心能力，强调可追溯、可验证。

当前版本优先解决“可用、可控、可扩展”的基础链路：

- 桌面端启动、工作空间管理、目录同步。
- 流式问答、思考过程展示、工具调用轨迹展示。
- 基于文件路径返回来源，避免无来源结论。

## 2. 为什么当前以 Markdown（MD）作为知识载体

当前阶段使用 MD 作为主要上下文载体，是一个刻意的工程取舍，而不是功能缺失：

1. 成本低、迭代快  
   不依赖额外数据库、向量库或索引服务，直接修改文档即可更新知识。

2. 透明可审计  
   内容完全可见，版本可对比，方便团队对提示策略和业务背景进行评审。

3. 与当前工具能力天然匹配  
   现有工具链本质是文件系统工具，读取 MD 的门槛最低、成功率最高。

4. 跨模型可迁移  
   MD 是通用文本协议，不绑定某个供应商特性，后续更换模型成本更低。

5. 便于分层组织  
   可以按“全局背景 / 业务规则 / 子模块说明 / 任务模板”拆分多个 md 文件，逐步演进。

## 3. 当前系统边界（给模型和开发者的现实约束）

- AI 不是直接操作 `source_path`，而是操作 `mirror_path`。
- 回答尽量基于工具读取到的文件，不鼓励凭空推断。
- 若引用文件内容，需要附来源路径，禁止伪造来源。
- 当前检索能力以关键词搜索为主，且保持不向量化、不引入 RAG。
- 当前会话记忆有长度限制，长期记忆尚未产品化。

这些边界意味着：  
“先把信息写进结构化 md，再让模型按需读取”是当前最稳定、最可控的方案。

## 4. 推荐的上下文文档结构（可直接照此扩展）

建议将项目知识拆成以下层级（可逐步补齐）：

- `electron/prompts/system-context.md`：全局背景、产品边界、架构共识（本文件）。
- `docs/domain/*.md`：业务域规则，例如工单、简历、财务、法务等。
- `docs/workflow/*.md`：任务流程模板，例如“分析 -> 修改 -> 验证 -> 回答”。
- `docs/constraints/*.md`：编码规范、安全边界、来源引用规范。
- `docs/glossary.md`：术语表，统一命名和口径。

## 5. 未来扩展路线

### 5.1 近端扩展（低改造成本）

- 在系统提示中增加“启动时优先读取核心上下文 md”的流程约束。
- 引入“上下文优先级”机制：全局 > 业务域 > 任务模板 > 临时消息。
- 对来源进行更严格的结构化输出（例如统一 `sources` 字段格式）。

### 5.2 中期扩展（能力增强）

- 增加结构化知识清单（metadata），例如文档版本、适用范围、生效日期。
- 引入轻量索引（标题/标签/关键词）以减少盲搜成本。
- 按 workspace 注入不同上下文包，实现“同一 MindWeave，不同项目人格”。

### 5.3 远期扩展（平台化）

- 强化文本检索与结构化索引（标题/标签/关键词/引用关系），仍保持不向量化、不引入 RAG。
- 增加规则编排层（策略路由、工具白名单、审批流）。
- 支持多 Agent 协同（规划 Agent / 执行 Agent / 审计 Agent）。

## 6. 为什么不是“只靠系统提示词一把梭”

超长系统提示词会带来三个问题：

- 可维护性差：改一个规则需要改代码并重新发布。
- 版本不可见：历史变更不透明，难以回溯。
- 上下文污染：所有任务都带全量信息，噪声高、成本高。

将核心认知外置到 MD，并让模型按需读取，是在当前阶段更稳健的工程方案。

## 7. 给模型的执行建议（可转写进系统提示词）

1. 优先判断是否需要读取上下文文档，不要直接凭经验回答。  
2. 先使用本文件中的规则，再执行任务。  
3. 若业务规则不足，再读取 `docs/domain` 与 `docs/workflow`。  
4. 输出结论时尽量附来源路径；不确定时明确说明假设。  
5. 修改文件前先说明意图与影响范围，降低误操作风险。
